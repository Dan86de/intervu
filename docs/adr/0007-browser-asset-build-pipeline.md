# Browser assets are real TypeScript, baked into a generated module for the ship and rebuilt from source in dev

## Status

accepted

## Context and decision

The annotation slice (#5) grows the browser side from a placeholder into real logic: an in-iframe SDK (`/sdk.js`) and a chrome controller (`/chrome.js`), plus the build-time Tailwind stylesheet (`/chrome.css`, ADR 0004).
Hand-written template-literal scripts - a few hundred lines of untyped, untested browser code embedded in server strings - were rejected.
The SDK and the chrome controller are authored as real TypeScript under `src/sdk/` and `src/chrome/` (plain DOM, no Effect) and bundled by `Bun.build` as self-running IIFEs; the stylesheet is compiled by the Tailwind v4 CLI.

intervu still ships as a single Bun-bundled file (`dist/intervu`), which has no source tree to build from at runtime.
So the three built assets are **baked into a generated module** (`src/generated/browserAssets.ts`) that `scripts/build-browser.ts` freezes before `scripts/build.ts` bundles `main.ts`; the `BrowserAssets` service statically imports it, so the strings are inlined into the single file.
Running from source, the same service instead **rebuilds the assets on daemon start** (`Bun.build` + the Tailwind CLI), so an edit to `src/sdk` or `src/chrome` is live on the next restart.
The branch is the presence of the SDK entry next to the module (`import.meta.dir`), which is true in a source checkout and false inside the bundle - no environment flag.

The browser sources are DOM TypeScript, so they typecheck under `tsconfig.browser.json` (with the DOM lib) rather than the DOM-free daemon config; only `src/sdk/protocol.ts`, the shared wire types, is free of DOM types so the server can import it.

## Considered options

- **Built TS baked into a generated module, dev rebuilds from source** - chosen: the browser code is real, typechecked, and testable; the single-file ship stays intact via a static import; dev keeps a fast edit loop without committing to a runtime build inside the shipped binary.
- **Hand-written template-literal scripts in server strings** - rejected: untyped, untested browser logic that the `Bun.build` pipeline exists to avoid.
- **Serve the assets from disk at runtime** - rejected: breaks the single-file ship; the daemon would need its source tree present.
- **Build to `dist/` and read at runtime** - rejected: the bundled binary cannot read sibling files reliably, and a static import keeps the assets inside the one file.

## Consequences

- `src/generated/browserAssets.ts` is a generated artifact (regenerated by `bun run build:browser`, which `bun run build` runs first); it is committed but never hand-edited, and excluded from Biome.
- The dev rebuild uses `Bun.build` and the Tailwind CLI, which exist only under the Bun runtime the daemon runs in. Tests therefore provide `BrowserAssets.bakedLayer`, never `layer`, so a unit test never triggers a build.
- A failed dev build fails the daemon's layer construction loudly through `BrowserAssetBuildError`; the shipped binary serves baked assets and never hits that path.
- `check:types` runs twice - the DOM-free daemon config and `tsconfig.browser.json` for the browser sources. Tailwind's entry stylesheet (`src/chrome/styles.css`) uses v4 at-rules Biome's CSS parser rejects, so it is excluded from Biome; the compiled output is what ships.
